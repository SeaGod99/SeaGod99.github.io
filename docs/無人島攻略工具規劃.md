## 1. 目標與定位

為**台服繁中玩家**做一套無人島開拓輔助工具。參考站「素素無人島」（<https://wrd.ffsusu.com/>）是簡中，繁中市場缺一個資料準確的版本。

三大功能（優先序）：

1. 素材／製作查詢（**原為優先 3，改列優先 1**，因為資料零卡點）
2. 動物時鐘（原優先 2，卡在動物名）
3. 開拓進度／解鎖表

**經濟型製作排程（工坊每週生產推薦）明確不做**：它依賴每週浮動的歡迎度／需求係數，等於再開一個「時尚品鑑等級」的週更負擔，違反 PROGRESS §三「優先做不需維護的純計算功能」。

## 2. 資料源：走 datamining-cn 的 CSV，不走 XIVAPI

XIVAPI v2 有 **40 個 `MJI*` sheet**，但**它的 schema 沒有為這些 sheet 命名欄位**——實測 `MJIAnimals` 只吐得出 `Icon`，`MJIRank`／`MJIItemPouch` 回傳空物件。

`thewakingsands/ffxiv-datamining-cn` 的 CSV 第 2 行帶欄位名、第 3 行帶型別與外鍵，是目前唯一拿得到完整欄位的來源。建置腳本 [`scripts/build-island.mjs`](../scripts/build-island.mjs) 走這條，並把原始 CSV 快取在 `out_data/mji-csv/`（47KB，可 `--offline` 重建）。

### 用到的表

| CSV | 列數 | 關鍵欄位 | 產出 |
|---|---|---|---|
| `MJIItemPouch` | 109 | Item, Category, Crop, Sort | `island-materials.json` |
| `MJIGatheringItem` | 48 | Item, Tool, X, Y, Radius, Map | 併入 materials 的 `gathering` |
| `MJIRecipe` ＋ `MJIRecipeMaterial` | 28 / 37 | KeyItem, ItemPouch, Material[], Amount[] | `island-recipes.json` |
| `MJIBuilding` ＋ `MJILandmark` | 15 / 10 | Material[], Amount[], Name, Icon | `island-buildings.json` |
| `MJIAnimals` | 43 | BNpcBase, Size, Rarity, Reward[0..1], Icon | `island-animals.json` |
| `MJIRank` | 20 | ExpToNext | `island-ranks.json` |
| `MJIDisposalShopItem` | 79 | Item(→Pouch), Count | `island-shop.json` |
| `MJIItemCategory` | 10 | Singular（簡中） | `island-categories.json` |
| `MJIName` | 6 | Singular（簡中，草原/溪流/森林/沙灘/山/洞窟） | `island-areas.json` |
| `MJICropSeed`／`MJIKeyItem`／`MJIGatheringTool`／`MJIText` | — | 被上面各表參照 | 併入 |

### ⚠️ 踩過的雷：row 0 不是空列

這組表裡 **`MJIItemPouch`／`MJIRecipe`／`MJIDisposalShopItem`／`MJIBuilding` 的 row 0（或子列 `0.0`）都是真資料**，不是慣例上的預設值空列。用 `key > 0` 過濾會**安靜地少掉**：無人島棕櫚葉、開拓用石斧、小島木屋 I。務必用「內容解析得出」判斷。

### ⚠️ 參照鏈不是直連 itemId

`MJIBuilding.Material[n]` / `MJIRecipe.Material[n]` **指向的是 `MJIItemPouch` 的 row 而不是 itemId**（配方還要再經一層 `MJIRecipeMaterial`）。解錯會拿到完全不相干的物品。

## 3. 繁中名策略（遵守專案鐵則）

| 類型 | 來源 | 狀態 |
|---|---|---|
| 素材／作物／畜產／成品／建築素材 | itemId → `data/items.json`（tw-items 台服官方名） | ✅ **100% 覆蓋** |
| 建築名（MJIText） | datamining-cn 只有簡中 | ⛔ 待人工補 |
| 素材分類名（MJIItemCategory） | 同上 | ⛔ 待人工補 |
| 島嶼地區名（MJIName） | 同上 | ⛔ 待人工補 |
| **動物名（MJIAnimals）** | **無來源，見下** | ⛔ **主要卡點** |

**簡中一律不簡轉繁。** 簡中字串放在 `nameCn` 欄僅供人工比對，`name` 一律留 `null` ＋ `nameMissing: true`，前端不顯示。

### 人工補名的機制

`data/island-names-tw.json` 是人工對照表（腳本會自動產生空白樣板）：

```json
{ "animals": { "1": "台服動物名" },
  "buildings": { "2": "台服建築名" },
  "categories": { "1": "台服分類名" },
  "areas": { "1": "台服地區名" } }
```

key 是各 `MJI*` 表的 row id。填好後重跑 `node scripts/build-island.mjs` 即合併進各檔的 `name`。

### 動物名為什麼拿不到（實測記錄，別重走）

`MJIAnimals` 只給 `BNpcBase`（14906–14948），要拿名字必須 BNpcBase → BNpcName → 台服名。兩條路都試過：

| 嘗試 | 結果 |
|---|---|
| `BNpcBase` → 本站 `monsters.json.baseId` | **7/43**，且對到的是**錯的**（「緊張的聲音」「薩瑞加的聲音」——baseId 撞號） |
| `BNpcBase` → Teamcraft `monsters.json` 反查 BNpcName → `tw/tw-mobs.json` | **0/43**（Teamcraft 只收有狩獵座標的 2333 隻，牧場動物不在內） |

**結論：只能從台服遊戲內人工抄錄 43 筆。** 這是一次性工作，抄完寫進 `island-names-tw.json` 即永久有效。

### 動物出現時段／天氣也不在 datamine

`MJIAnimals` 沒有時段與天氣欄位——那是 BNpc 的 spawn 條件，不在任何簡單表裡。這就是為什麼素素無人島那類站是**社群觀測**來的。要做動物時鐘，這 43 筆的 `spawn` / `weather` / `coords` 同樣得人工補（欄位已在 schema 裡留好）。

### 底圖與座標（2026-07-23 已解決並驗證）

**無人島底圖就在既有的 `maps.json` 裡**——`id 772`，名稱是「無名島」（`nameEn: "Unnamed Island"`），region「？？？？」、type `instance`。先前沒被認出來，是因為它不叫「無人島」。

- **底圖已下載**：`assets/maps/h1m2_01.jpg`（608KB，`node scripts/download-maps.mjs --id 772`）
- **XIVAPI 有三張變體**：`h1m2/01` 是**開拓後、含村莊建設的完整版**（`maps.json` 指向的就是它）；`h1m2/02`／`03` 的西南半島是未開拓地形，不採用
- **天氣表已備**：碧空 25%／晴朗 45%／陰天 10%／小雨 10%／薄霧 5%／暴雨 5% ——**動物時鐘要用的天氣機率就在這張地圖的 `weatherRates` 裡**，不必另尋來源

#### 座標換算（已驗證）

`MJIGatheringItem` 的 X/Y 是**世界座標**（實測 X −248~765、Y −694~246），不是遊戲內地圖座標。換算走 FFXIV 標準式：

```
c    = sizeFactor / 100                        // 無人島 = 1
frac = ((world + offset) * c + 1024) / 2048    // 0..1，可直接乘底圖寬高
game = frac * 41 / c + 1                       // 遊戲內座標 1..42
```

`offsetX = −175`、`offsetY = 138`。換算後採集點落在 X 13.0~33.3、Y 10.4~29.2。

**驗證方式（不是憑公式相信）**：把 48 個採集點畫上底圖——

- ✅ **全部 48 點落在陸地上**，沒有一個在海裡
- ✅ `mapLayer = 1` 的 9 筆（石炭／堆積岩／燈火茸／氣泡水／幻影石／黃銅礦／金礦／鷹眼砂／水晶層，全是洞窟產物）**緊密聚在東北山區的洞窟位置**
- ❌ 對照組：把 Y 反轉重畫，有點掉進海裡、洞窟點也散開 → 確認目前方向正確

`island-materials.json` 的 `gathering.coords` 已改存**站內標準格式 `{mapId, x, y}` 的遊戲內座標**（`mapId: 772`），可直接餵給既有的地圖元件；原始世界座標保留在 `gathering.raw` 供日後校驗。

> 底圖的標題文字在 XIVAPI 匯出的圖上看起來像鏡像，但那只影響觀感、不影響幾何——幾何已用上面的落點測試確認。

#### ⚠️ 一個區域可以採到好幾種素材

`MJIGatheringItem` 是**一素材一列**，很容易誤以為「一種素材＝一個採集點」。實際上多個素材共用同一組座標＋半徑：**48 種可採素材只分佈在 26 個區域**，最多的一區有 4 種（如 X:33.3 Y:24.6 可採球藻／珊瑚／烏賊／水母）。

因此資料層把「區域」升為一級概念（`island-gather-areas.json`），素材上掛 `gathering.areaId` 連回去。**前端要以區域為單位呈現**，不要一種素材畫一個圈——那會同時誤導兩件事：以為只有一個點可採、以及以為那個點只產一種東西。

#### `mapLayer` 的語意（已確認）

`MJIGatheringItem.Map` 只有 0/1 兩種值，而島嶼地區（MJIName）有 6 個 → 它**不是地區**。實測 `1` 的 9 筆全是洞窟產物 → **0 ＝ 地上、1 ＝ 洞窟**，已加 `layerName` 欄位。

### 資料層的已知未解項

- 48/109 素材有採集資訊，其餘為作物／畜產／製作產物（本來就不是採集來的）。
- 素材的**地區歸屬**（草原／溪流／…）datamine 沒有直接欄位；不過座標既已驗證可用，前端直接以地圖標點呈現即可，地區分類非必要。

## 5. 功能規劃

### 5.1 素材／製作查詢（優先 1，零卡點）

**資料**：`island-materials` ＋ `island-recipes` ＋ `island-shop`（＋ 既有 `recipes.json` jobId −10 的 106 筆經濟型製作）。全部現成。

**呈現**：

```
[製作清單] [素材反查] [採集地圖]      ← 三個分頁

製作清單  卡片＝成品，展開看素材×數量；篩選：人工製作／經濟型製作、解鎖順序
素材反查  選一素材 →「怎麼拿」＋「用在哪」
          ┌ 無人島石灰岩 ────────────────┐
          │ 採集：X:17.6 Y:23.3（地上）    │
          │ 收購：3 貝幣                  │
          │ 用於：開拓用石錘、屯貨倉庫 Lv2…│
          └──────────────────────────────┘
採集地圖  島嶼底圖（assets/maps/h1m2_01.jpg）＋48 個採集物標點，可切「地上／洞窟」兩層
```

### 5.2 動物時鐘（優先 2，卡在人工資料）

**資料**：`island-animals` 已有體型／稀有度／掉落／圖示；**缺名稱＋出現時段＋天氣＋座標**（43 筆人工）。

**呈現**：跟釣魚頁**同一個形狀**，可直接沿用 `fishStatus()` 的開窗計算與 [`assets/js/eorzea-weather.js`](../assets/js/eorzea-weather.js)：

```

├ [體型 小/中/大] [素材反查] [狀態 出現中/未出現] ┤
│ 🐑 動物名        ● 出現中 · 剩 12:34            │
│    中型・捕獸繩   素材：牧場動物的獸毛／角片      │
│    ▓▓▓░░░░ ET 時間窗 bar                        │

```

預設排序＝開窗時間（出現中優先），與釣魚頁一致。43 筆一次畫完，不需分頁。

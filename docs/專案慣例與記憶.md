## 1. 溝通與工作習慣

### 1.1 一律用繁體中文回覆
所有對使用者的說明、摘要、選項、驗收報告一律**繁體中文**，不要夾雜英文句子／標題（曾出現 "Good, preserved" / "Bad, fixed" 被指正）。技術名詞（class 名、函式名、指令、CSS 屬性、commit 前綴如 feat/fix）可保留原文，但敘述句一律繁中。
**為何**：使用者為 FFXIV 台服（繁中）玩家，母語繁中。

### 1.2 終端機用 VSC 內建終端機
下指令時走 VS Code 內建終端機，**避免使用者電腦一直彈出獨立終端機視窗**。

### 1.3 多行 commit／PR 訊息走訊息檔 `-F`
用 **Bash 工具**下 `git commit` 多行訊息時，**不要用 PowerShell here-string `@'...'@`**——Bash 是 POSIX sh，會把 `@` 當字面字元，導致訊息開頭／結尾多出 `@` 行（曾犯兩次都得 `--amend` 修）。
**正確做法**：先用 Write 把訊息寫到 scratchpad 的 `.txt`，再 `git commit -F <path>`（或 `--amend -F <path>`）。本機 CRLF 環境穩定、也避開引號跳脫。

### 1.4 gstack 技能組
本站已裝 gstack 技能組，各使用情境對照表見 [CLAUDE.md](../CLAUDE.md)。常用：`/browse`（開頁截圖驗收）、`/qa`、`/verify`、`/design-review`、`/code-review`、`/ship`、`/scrape`。

## 2. 收藏／追蹤頁共通慣例

適用範圍：`/collections/*`、`/minions/`，以及具「追蹤擁有進度」性質的工具頁。

### 2.1 取得方式＝最優先資訊，永遠預設顯示
使用者明確要求：收藏／追蹤頁「最想知道的是**怎麼取得**」，因此取得方式**預設顯示且視覺優先**，不可只靠 hover、不可藏在展開鈕後（舊的點擊展開 `collection-detail.js` 已刪）。
- 各明細卡頁的 `.X-detail` 區塊永遠 `display:block`；金色「取得方式」標籤（`::before`）＋金色左側強調線；空內容 `:empty{display:none}`。
- emotes 同款（置中版型、無左線）；hairstyles 來源本就常駐標籤。
- minions 是密集圖示格：名稱下方加一行金色**來源類別**（永遠顯示），完整細節點 ⓘ 釘選 tooltip。
- **勿改回 hover／toggle。**

### 2.2 共用 CSS：`assets/css/common.css`
收藏類頁面的共用 CSS 已抽成 `assets/css/common.css`（2026-06 重構）。內含暗色 `:root` 設計變數、版面外框（header/toolbar/filter/progress/footer）、`.col-*` 卡片系統（col-grid / col-card / col-icon / col-name / col-tags / col-detail / src-tag / type-*）。
- 各頁 `<head>` 順序：先 `<link common.css>` → 該頁 `<style>`（覆寫勝出）→ 最後 `<link theme.css>`（亮色由 `[data-theme=light]` 特定度勝出）。
- 每頁只在自己精簡 `<style>` 設 **4 個強調色變數**：`--accent`（用**暗色字面 hex**，例 #a78bfa，使 common 的 `color-mix(... N%)` 等同原 `rgba(...,0.N)` 像素一致）、`--accent-2`（進度條漸層尾色）、`--glow-1`／`--glow-2`（背景光暈）。
- **採用 .col-* 的頁**：mounts、emotes（置中）、barding、orchestrion、triple-triad、blue-magic。
- **保留專屬卡片、只共用外框的頁**：hairstyles（橫向卡）、minions（小方格＋tooltip）。
- 新增收藏頁：link common.css → 套 .col-* 卡片 → 設 4 個 accent 變數，專屬樣式才寫進該頁 `<style>`。

### 2.4 版本篩選統一格式「N.x 資料片繁中名」
2026-07-02 定案（commit bbcd9f1），取代先前混用的四種寫法。
- 六個資料片：**2.x 原初之地／3.x 蒼天之禁地／4.x 紅蓮之狂潮／5.x 暗影之逆焰／6.x 曉月之終途／7.x 金曦之遺輝**。
- 篩選控制項（chips/select）顯示一律「N.x 資料片名」（例 `3.x 蒼天之禁地`）；**資料值**保持資料片繁中名或 major 數字不變，只在顯示層加前綴（樂譜／藏寶圖以 `EXP_PREFIX` 對照表轉顯示）。
- 篩選邏輯以 major（`parseInt(patch,10)`）對應資料片，不逐小版本開 chip；篩選區標題統一「版本」。

- 已套用：mounts、hairstyles、orchestrion、exploration-log、treasure-maps（列標題）、weather（地圖群組）、fishing、gathering、gathering-log、aether-currents；寵物／幻卡／鳥鞍也已補上（PATCH_BANDS 模式同坐騎頁）。

### 2.5 收藏頁共用引擎 `assets/js/collection-tracker.js`（2026-07-22 建立，07-23 擴充）
§2.3 的標配功能全部收進單一引擎，取代各頁 ~480–690 行的重複 JS。**全部 12 個追蹤頁**都已改為「header＋一份 `CollectionTracker.init({...})` 設定」：mounts／minions／barding／orchestrion／emotes／hairstyles／blue-magic／triple-triad（07-22）＋ exploration-log／aether-currents／gathering-log／fishing（07-23）。**不要再手寫追蹤頁的篩選／排序／渲染迴圈。**
- **引擎負責**：資料載入＋PatchGate 過濾、進度條＋`ffxiv_snap_<snapKey>` 首頁快照、工具列（搜尋／擁有切換／排序／批次標記／匯入匯出／清除）、標籤篩選區、格線渲染、鍵盤＋ARIA、以及 **URL 狀態同步**（`?q=&own=&sort=&f_<id>=`，打字 replaceState、離散操作 pushState、popstate 還原，可分享／加書籤）。控制面渲染進掛載點 `<div id="ct-root"></div>`。
- **各頁只提供設定**：`dataUrl／metaUrl／storageKey／snapKey`（＋可選 `schema／fileBase` 保留既有匯出檔相容）、`verb`（擁有/解鎖/習得）、`unit`（隻/個/張…）、`noun`；資料 hook `include(e,gp)／keyOf(e)／alwaysOwned(e)／searchText(e)／prepare(list)`；`filters:[{id,label,options(list),match(e,v)}｜{kind:'patch'}]`；`sorts:['name','patch-desc','patch-asc',{value,label,cmp}]`；`card(e)->innerHTML`；互動 hook `onCardClick(e,ev)`（回傳 true＝不切換擁有，如市場連結、幻卡 📍）、`onCardCreate(e,cardEl)`（掛額外互動，如寵物 hover 提示框）；版型 `gridClass／cardClass`（預設 col-grid/col-card；hairstyles=hs-*、minions=minion-*）。
- **搜尋預設涵蓋 `sources[].detail`**（可搜取得方式）；各頁可覆寫 `searchText` 加更多欄位（如幻卡加 npcName／mapName、青魔加習得地點）。
- **相容性鐵則**：`storageKey`／`schema`／`keyOf` 必須沿用各頁原值，否則洗掉使用者既有進度。特例：**minions 的 `keyOf` 回傳純數字 id**（既有存檔是 `[1,2,3]` 非 `['id:1']`）；emotes/hairstyles 等的 schema 沿用原字串。
- **驗證方式**：本機 headless Chromium 在此環境無法啟動（`chrome-headless-shell` 崩潰，exit 3221225786），改用 **jsdom**（暫存區裝，不進 package.json）載入引擎＋各頁真實設定＋mock fetch 讀真實資料檔做 DOM 層驗證。新增/改引擎後照此跑一次。
- **新增收藏頁**：複製任一遷移頁的 head＋設定即可；不要再回頭手寫 filter/sort/render 迴圈。

**三個選用能力（2026-07-23 新增，不設定即完全維持原行為）**
- **`subsOf(entry) -> [子項目]`「子項目模式」**：追蹤單位不是卡片本身而是卡片裡的東西時使用——風脈泉頁一張卡＝地區、要打勾的是 303 個風脈泉；採集紀錄頁一張卡＝採集點、要打勾的是產物。設定後 **`keyOf` 收到的是子項目**、進度分母＝子項目總數、卡片 `.owned` 代表「該卡子項目全數完成」、批次標記作用於篩選結果的所有子項目。**單位依 `keyOf` 去重**（同一件產物出現在多個採集點只算一件）。此模式下引擎**不綁卡片層級的點擊切換**，各頁在 `onCardCreate` 裡自己綁子項目並呼叫 `tracker.toggleKey(key)`。
- **`pageSize`（分頁）**：資料量大的頁面用（釣魚 60／採集點 40）。頁碼同步到網址 `?p=`（1-based，可分享），任何篩選／搜尋／排序變動自動回第 1 頁，頁碼超出範圍會夾回。樣式在 common.css 的 `.ct-pagination`／`.ct-page-btn`。
- **`onRender(list, pageSlice, tracker)`**：每次重畫格線後呼叫，讓各頁同步自己的附加檢視（地圖標點、目標魚面板）。

**其他選用設定**：`rowsOf(json)`（資料不在 `data[]` 時指定，風脈泉庫是 `zones[]`）、`defaultSort`（釣魚頁預設 `window`＝依開窗時間）、`exportExtra()`／`onImport(data)`（頁面自有的額外進度跟著匯出匯入走，如釣魚的目標魚清單）。匯入已改為同時接受 `owned`／`unlocked`／`done` 三種舊鍵名。

**踩過的雷**：卡片內若有自己的點擊行為（釣魚頁的 🎯目標／📍地圖／點圓圈勾已釣／點其他開詳情），**不要另外在 `#ct-root` 上做事件代理**——引擎已經綁了卡片層級的 toggle，兩邊都處理會變成「切換兩次＝沒切換」。正解是用 `onCardClick(e, ev)` 分流，回傳 `true` 表示「本頁已處理、不要切換」。

### 3.3 時間一律 24 小時制（2026-07-24 定案）
ET 時窗、天氣時段、現實時刻，**顯示與儲存全部用 24 小時制**（`ET 09:00–12:00`、`7/25 14:26`），不用 am/pm。

**Why**：`12am`／`12pm` 誰是午夜誰是中午極易看錯。無人島動物的外部來源就栽在這裡——原文寫
`9am-12am`（字面是 09:00→00:00、15 小時），實為 `09:00-12:00`。是「全表時窗皆 3 小時」這個結構性
特徵才把它抓出來的，光看字面看不出錯。

**How to apply**：
- 資料檔存 0–23 的整點，時窗用半開區間 `[start, end)`。
- 外部來源寫 am/pm 時**先換算再填**，不要把 am/pm 原樣留在資料裡。
- 能加護欄就加：`build-island.mjs` 會驗 `animalSpawns.start/end` 必須是 0–23 整數、必須成對、不得相等，
  寫成 `"9am"`／`9.5`／`25` 都會 throw 並指出是哪一筆。**資料檔寫錯就炸**這種護欄比文件約定可靠得多
  （天氣名也是同樣做法，見 §5.9b）。

### 4.3c 社群來源要「對回遊戲資料」才算數（2026-07-24 定案）

鐵則是「台服官方優先、**社群繁中站次之**」——但「次之」不等於照抄。使用者給社群來源時（例：無人島動物來自
[steamxo](https://www.steamxo.com/2023/01/05/2700775)，附註「雖然是無官方確認，但資料應該是正確的」），
**先做結構性對位，別直接寫進資料檔**。無人島動物那次的三層驗證可以當範本：

1. **用來源沒給的欄位當對位鍵**。文章只有名稱／體型／時間／天氣／座標，沒有 row id。
   拿「體型＋稀有度＋兩項報酬素材的**先後順序**」對回 `MJIAnimals`，29 筆中 27 筆唯一命中。
   關鍵是**稀有度不是文章給的**，是從「常駐／與常駐同座標／座標獨占」推出來的 → 等於一條獨立約束。
   對位率高本身就是「這份資料是真的」的證據；對不上的那幾筆才是要查的。
2. **歧義用另一個維度解，不要猜**。奧猴／松鼠的報酬完全相同 → 抓 `Icon` 圖檔下來**看圖**判定
   （70123 靈長類大眼＝奧猴、70127 尖吻齧齒＝松鼠）。這比「哪個 sort 比較小」之類的推測可靠。
3. **名稱另找站內官方佐證**。29 個名稱的生物詞幹全部能在 `data/monsters.json`（`nameSource=tw-mobs`
   台服官方怪物名）找到同詞條 → 證明不是簡轉繁也不是臆測。

**外部來源會有錯，對位就是為了抓出來**：那篇有三處錯誤（時窗筆誤、兩欄對調、兩筆報酬與 datamine 不符）。
原則是**datamine 有的欄位一律以 datamine 為準**，外部來源只補 datamine 沒有的東西。
來源網址、等級（官方／社群）、驗證方法、已知錯誤全部寫進資料檔本身（見 `data/island-names-tw.json` 的
`_animalSource`），別只留在 commit message——換機之後只有 repo 內的東西跟得走。

**譯名 vs 數值要分開判斷**：動物「名稱」受譯名鐵則約束；但「出現時段／天氣／座標」是**語言中立的數值**，
任何可靠來源皆可，驗證方式是對回遊戲資料而非比對譯名。卡在譯名時先問一句「我缺的到底是名字還是數字」。

### 4.3d 例外：非物品的敘述字串可簡轉繁，但要標記且會出錯（2026-07-24）

使用者 2026-07-24 就無人島的**製作類型／建築／地區／分類**名指示「直接翻譯成繁體就好，大致上的文字應該是一樣的」。
這是使用者的決定，照做——但實作方式有講究：

- **只開給非物品字串**。凡是拿得到 `itemId` 的一律走 `items.json` 的台服官方名，**不因為這個口子開了就一起轉**。
- **轉出來的名字要標記來源**。`island-names-tw.json` 的 `_nameSourceBySection` 把各區塊標成
  `cn-hant`／`community-tc`／`tw-official`，`build-island.mjs` 寫成每筆的 `nameSource`，
  前端以虛線底線＋tooltip 標注「非台服官方譯名」。簡中原文保留在 `nameCn`，日後抄到實際用字可直接替換。
- **它確實會錯，而且錯得很像真的**。我逐字把 `开拓工房` 轉成「開拓工房」，但台服官方物品叫
  「**開拓工坊**酸泡菜／烤南瓜／水煮蛋」——是**工坊**。這是靠反查 `items.json` 才發現的。
  物品名的實例更多：`开拓用鹤嘴锄`→**開拓用十字鎬**、`高级捕兽用催眠球`→**高級捕獸用睡眠球**、
  `海岛红辣椒`→**海島紅彩椒**、`海岛西葫芦`→**海島櫛瓜**、`海岛西兰花`→**海島花椰菜**、
  `长臂虾`→**無人島長臂蝦**。全都不是換字能得到的。
- **能加護欄就加**。`build-island.mjs` 會檢查開拓等級解鎖敘述裡以「」括起來的詞是否為
  `items.json` 的台服官方物品名，不是且不在 `NON_ITEM_TERMS` 白名單就 **throw**——
  這擋的是「日後有人把簡中攻略整段貼進來」。已實測攔截。

**通用做法**：轉繁前先拿候選詞去 `items.json` 全文搜一遍，官方物品名裡出現過的用字就是官方用字。
這一招在本次抓到「工坊 vs 工房」，成本極低，值得每次都做。

### 4.4 天氣資料管線（重大 bug 教訓）
maps.json 的 `weatherRates` 由 `scripts/fix-mapkeys.mjs` 從 XIVAPI v2 回填。兩個關鍵事實（2026-07-02 修復）：
1. **XIVAPI v2 的 `WeatherRate.Rate[]` 已是「各段機率」**（加總=100），**不是**原始 sheet 的累進上界。當成累進相減會產生負值被丟棄，曾造成 66/67 張地圖天氣表殘缺（加總只有 30–45%）。
2. **天氣槽位順序不可合併同名天氣**——天氣種子演算法（`assets/js/eorzea-weather.js`）靠順序累加出 seed 區間，合併會位移邊界算出錯誤天氣。
- **天氣譯名唯一權威表**＝fix-mapkeys.mjs 的 `WEATHER_TC`（微風=Wind、強風=Gales、暴雨=Showers、雷雨=Thunderstorms、星磁暴=Astromagnetic Storms、月塵=Moon Dust）。`eorzea-weather.js` 的 `WEATHER_INFO` 名稱已對齊；**改任一邊必須同步另一邊**，否則 NAME_TO_KEY 反查斷裂、限天氣魚永遠比對不到。
- `fishes.json` 的 `weatherSet[].name` 是英文名，釣魚頁用 `EN_TO_KEY` 轉引擎 key 後比對。31 種魚 spotId=null 無法算天氣窗口。
- **`eorzea-weather.js` 的演算法已用獨立實作交叉驗證過（2026-07-24）**：與社群通用的 SaintCoinach／`WeatherFinder`
  版本比對 **22000 個天氣時段，seed 與天氣結果皆 0 筆不一致**，無名島（map 772）機率表也逐項吻合。
  **不必再懷疑這支的正確性**；天氣算錯時先查 `weatherRates` 資料與 mapId，不要重寫演算法。

### 5.9 幻化配裝圖鑑（glamour）：取得方式一律吃 `item_sources.js`，不要信單一 `source` 欄
`curated_outfits.js`／`mirapri_outfits.js`／`official_sets.js` 裡每件裝備的 `source`／`src` 欄**只是「其中一種」取法**——`pipeline.py` 的 `_best()` 取優先度最高那個、`build_sets.py` 的 `fmt_piece_source()` 取前兩條且多副本只寫第一個＋「等N處」。sources.json 有 26% 的裝備是多來源，照這些欄位做篩選／統計會少掉約四成的取法（2026-07-19 使用者回報「明明拿得到卻篩不到」）。
- **完整清單＝`tools/glamour/item_sources.js`**（`build_item_sources.py` 產生，裝備 ID → 正規化來源鍵）。前端 `pieceSrcKeys()`／`entrySrcKeys()` 已封裝好，**任何新功能要用取得方式一律走這兩個函式**，別再讀 `p.source`。
- 來源鍵不含價格／NPC 地點／副本類型，目的是讓官方套裝與配裝兩種寫法收斂成同一個選項；要顯示細節才用原本的 `source`／`src` 字串。
- **重建任何一份前端 js 後必須重跑 `build_item_sources.py`**（`update_all.py` 的 full／local 都已排在 `build_site.py` 之後）。漏跑不會報錯，只會安靜地退回單一來源。
- **頁面狀態同步到網址（History API）**：檢視／篩選／細項／搜尋／排序／頁碼／開啟中的套裝／語言都寫進 query string，瀏覽器上一頁才有東西可回，網址也能分享。**改任何會改變畫面的互動，記得補 `syncURL(true)`**；打字搜尋、彈窗內上下套、關彈窗要用 `syncURL(false)`（replace），否則歷史會被灌爆、或「關掉彈窗後按上一頁又開回來」。兩個踩過的雷：① **Bootstrap modal 在開啟動畫進行中會忽略 `hide()`**（點開後馬上按上一頁關不掉）→ 用 `modalShowing`／`closeAfterShown` 旗標等 `shown` 事件再關；② 判斷彈窗開著**不能看 `.show` class**（非同步才加上去），要用自己維護的 `openModalId`。
- **精選套裝的裝備以 `iid`（道具 ID）為權威**：`data/curated_outfits.json` 每件都已回填 `iid`（2026-07-20，500/500），`build_site.py` 的 `stamp_badges()` 有 id 就用 id，只有沒 id 的新件才退回名稱比對。**原本純靠名稱猜的問題是「猜不到不會報錯」**——該件靜默掉 iid，連帶失去徽章與 `item_sources.js` 的取得方式；改版撞同名或官方改譯名就會踩到。新增套裝時請直接填 `iid`（查名稱時本來就會拿到），或事後跑 `backfill_curated_iid.py --apply`（撞名會拒填並列報告，不亂猜）；`health_check.py` 已會擋缺 id 的件。
- **精選的 `zh`／`ja`／`en`／`patch`／`lv` 一律由 `iid` 從 DB 推導，不手填**：2026-07-20 用 iid 逐件稽核 500 件，抓出 **211 處人工輸入誤差**——版本一律填 7.0（實際 7.3/7.4/7.5）、等級多半留在預設 Lv.1（152 件）、日文名濁點長音抄錯（`バルチザン`→`パルチザン` 等 18 件）、英文名由另一語言回譯（19 件），最嚴重的是**台服未實裝卻填自編繁中名 11 件**，害 5 套被誤標「🇹🇼 繁中版可幻化」（違反 §4「繁中名絕不自己翻」）。對照組：社群配裝 33705 件同樣稽核只有 0.1% 不符、日文名 0 不符——**問題全出在人工輸入，不是資料來源**。已由 `build_site.py` 的 `apply_db_fields()` 每次建置重算，並有 `normalize_curated_from_db.py` 做一次性校正。
- **`job`（職業限制）也由 `cjc` 推導，不手填**：2026-07-20 稽核出手填 job 一堆錯——自創詞「偵察職業」（cjc 103 ROG NIN VPR，正解＝忍者、劍蛇師）、把整職能寫成單一職業、只列部分職業、專武填錯職能（月讀太刀填「盾衛職業」實為暗黑騎士專武 DRK）。`build_site.job_from_cjc()` 是唯一權威：整職能→群組名、真子集→列具體職業、全戰鬥→全職業、Disciple of Hand/Land→製作/採集。cjc id→職業字串在 `data/xivapi_sets_cache.json` 的 `cjc_names`（build_sets.py 產出）。前端 `jobCodes()` 會把「忍者、劍蛇師」用 `、` 拆開查 JOB_CODES，所以顯示字串換法不影響繁中版可幻化判定。
- **「部位對不上」是內容錯，不能自動修**：#01 腳部原本填了腿部道具「寄葉五五式禦敵軍褲」。看圖判讀後確認腿部已被「男爵及膝褲」佔用（染色標籤煤玉黑／羅蘭莓與圖上紅裙黑腰帶吻合），圖上那雙過膝長靴是同系列的**「寄葉五五式禦敵軍靴」(33566, YoRHa Type-55 **Thighboots**)**——投稿者標的是同系列的「脚衣」而非「軍靴」。這類錯誤 `health_check.py` 現在會擋（比對 DB 部位），但**修正一律要看圖，別照名稱猜**。
- **取得方式的分類按鈕與細項下拉必須吃同一份來源**（都走 `entryStSet()`／`entrySrcKeys()`）。2026-07-19 曾因分類只看整套那一種 st、細項看完整鍵而互斥：「夢幻套裝」整套來源是 🗓️Starlight Celebration（st=event）但件是 🪙夢幻帽布料 兌換，同時選「代幣兌換」＋該細項會篩出 0 筆。改法是分類也取完整來源鍵的 st 聯集，讓「細項 ⊆ 分類」恆成立（已用全鍵稽核驗過 0 例外）。卡片上的 tag 徽章仍顯示主要來源，不跟著放寬。
